Knowledge Base Builder: Multi-Source Knowledge Base Construction with Large Language Models

Kostadin Devedzhiev
Centre for Human-Inspired AI, University of Cambridge
kgd26@cam.ac.uk
Abstract

We present knowledge-base-builder, a Python package that consolidates heterogeneous research artifacts—publications, code repositories, web documentation, lecture recordings, and presentation materials—into structured Markdown knowledge bases using large language models. The package ingests content from 11 source types and produces output suitable for retrieval-augmented generation (RAG) systems, vector database preprocessing, and web-crawlable context files. We evaluate the pipeline across four experiments examining source type fidelity, scaling behavior, temperature sensitivity, and chunk size impact, finding that structured sources achieve 100% keyword recall, temperature controls lexical diversity without affecting factual coverage (Jaccard: 0.87 at t=0 vs. 0.46 at t=1), and output length saturates beyond 4 sources.

1. Statement of Need

Retrieval-augmented generation has emerged as the dominant paradigm for grounding LLM outputs in domain-specific knowledge [Lewis et al., 2020] [Gao et al., 2024], yet the quality of RAG systems is fundamentally constrained by the quality of their input knowledge bases. Researchers routinely need to synthesize knowledge from sources that span formats, platforms, and modalities—arXiv preprints, conference slide decks, GitHub codebases, supplementary Jupyter notebooks, and blog posts—each requiring different extraction tools.

This fragmentation creates three problems. First, reproducibility suffers when knowledge base construction depends on undocumented scripts—violating FAIR principles [Wilkinson et al., 2016]. Second, coverage is limited because researchers default to PDFs, omitting code, videos, and structured data. Recent work shows LLMs can extract structured information from scientific text with high fidelity [Dunn et al., 2024], but orchestrating this across source types requires significant effort. Third, iteration is slow because any source change requires rebuilding from scratch.

knowledge-base-builder provides a declarative pipeline: specify sources as a dictionary of URLs and paths, and the package handles format detection, concurrent extraction, deduplication, LLM structuring, and output generation. A metadata sidecar records provenance; incremental builds allow efficient iteration.

Output Formats

  • Markdown — structured knowledge base with hierarchical headings
  • llms.txt — llmstxt.org spec-compliant navigation + full-text companion file
  • Chunks — JSON array with per-chunk metadata for vector DB ingestion (no LLM costs)
  • Metadata sidecar — .meta.json with provenance, timing, and quality scores

Structured sources (PDF, GitHub) achieve 100% keyword recall, while HTML-based sources score 83–92%. The pipeline acts as a formatter for already-structured content and a lossy compressor for unstructured web content. Sitemap crawling shows the lowest reproducibility (Jaccard 0.55).

Output length saturates at 4 sources—words per source drops from 159 to 56 as more sources are added. The single-LLM-call architecture imposes an implicit output ceiling. For large corpora, the chunks output mode (bypassing LLM summarization) is recommended.

Key Result

Jaccard consistency drops monotonically from 0.87 to 0.46 as temperature increases, while keyword recall remains stable at ~73%. Temperature controls lexical diversity without affecting factual coverage. This finding generalizes to any LLM-based summarization pipeline [Dunn et al., 2024]. Researchers requiring reproducible KBs should use temperature 0.0.

The 40K character chunk size achieves the highest recall (81%). Very small chunks (5K) incur a 2x time penalty from additional LLM calls. Very large chunks show no quality improvement, suggesting that the default 80K setting is safe but suboptimal.

7. Research Impact

The package addresses a concrete bottleneck in LLM-augmented research: consolidating scattered artifacts into structured, machine-readable knowledge bases. The arXiv and Jupyter integration enables a workflow where a researcher points the tool at paper IDs and companion code repositories to produce a structured literature review—a task that previously required manual curation across platforms. This supports the systematic review automation goals identified by [Ofori-Boateng et al., 2024] and aligns with emerging context engineering practices [Qian et al., 2025].

The empirical findings are generalizable. The monotonic relationship between temperature and lexical consistency (Jaccard 0.87→0.46, recall stable) applies to any LLM summarization pipeline. The output saturation beyond 4 sources informs architectural decisions for RAG preprocessing at scale [Gao et al., 2024]. The metadata sidecar supports FAIR principles [Wilkinson et al., 2016] by recording complete build provenance.
